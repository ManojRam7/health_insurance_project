## 1. Context & Objective

The Members Silver layer transforms **raw member records** into a **clean, validated, policy-linked business-ready dataset** that:

- 👤 Creates a single, trusted member dimension  
- ✅ Enforces member identity & policy consistency  
- 📊 Adds demographic insights & member-level metrics  
- 🔒 Provides data quality transparency  
- 🎯 Enables 360° customer analytics  

**Enterprise standards applied:**
- Schema enforcement  
- Primary key validation  
- Foreign key validation (Policy_ID)  
- Demographic sanity checks (age, BMI)  
- Contact data validation (email, phone)  
- Address standardization  
- Categorical normalization (gender, status)  
- Feature engineering (age, tenure)  
- Data quality flags  
- Deduplication  
- Quarantine-first approach  
- Metrics logging

## 2. Bronze vs Silver Members

| Area | Bronze Issues | Silver Solution |
|------|---------------|-----------------|
| **Schema** | Raw CSV types | Enforced enterprise schema |
| **PK (Member_Key)** | Unvalidated | Strictly validated, duplicates removed |
| **Demographics** | Unchecked age, gender | Validated & flagged |
| **Contact Data** | No validation | Email/phone format checks |
| **Address** | Raw text | Standardized & validated |
| **Policy FK** | Unvalidated | Linked to Silver Policies |
| **Age Metrics** | None | Member_Age derived from DOB |
| **Tenure Metrics** | None | Tenure_Days calculated |
| **Status** | Messy variants | Standardized (Active/Inactive/Unknown) |
| **DQ Signals** | None | 4 flags (age, contact, address, status) |
| **Quarantine** | N/A | Bad records isolated |

**Result:** Single, consistent, trusted member dimension for 360° customer analytics.

## 3. Transformations: Bronze → Silver

### Read & Enforce Schema

Cast columns to enterprise schema:
```
Member_Key, Member_ID, First_Name, Last_Name,
Date_of_Birth, Gender, Email, Phone,
Address, City, Postcode, Member_Status,
Policy_ID, Member_Registration_Date
```

### PK Validation

`Member_Key` must not be NULL → otherwise quarantined.

### Deduplication

Keep latest record per `Member_Key`:
```
drop_dupes_keep_latest(["Member_Key"], ["Member_Status"])
```

### Policy FK Validation

- Check: `Policy_ID` exists in Silver Policies  
- If NULL or invalid → quarantine with code `FK_POLICY_INVALID`  
- **Business impact:** Ensures members are policy-linked  

### Demographic Validation

**Age validation:**
- Calculate age from Date_of_Birth  
- Age must be between 0 and 150  
- If DOB missing, age = NULL  
- Flag invalid ages via `dq_age_valid` (1 = valid, 0 = implausible)  

**Gender validation:**
- Standardize to {M, F, Other} or NULL  
- Invalid values → set to Unknown (not quarantined)  

### Contact Data Validation

**Email validation:**
- Basic format check (contains @, domain)  
- Flag invalid emails via `dq_contact_valid` (1 = valid, 0 = invalid format)  

**Phone validation:**
- UK phone format expected  
- Basic sanity (11 digits, +44 prefix, etc.)  
- Flag invalid via `dq_contact_valid`  

### Address Standardization

- Trim and uppercase  
- Validate postcode format (UK postcodes: letters & numbers)  
- Flag invalid addresses via `dq_address_valid` (1 = valid, 0 = invalid)  

### Status Normalization

Standardize to controlled list:
```python
Member_Status_Clean = CASE
  WHEN Member_Status IN ('Active', 'Inactive', 'Lapsed') THEN Member_Status
  ELSE 'Unknown'
END
```

Flag status validity via `dq_status_valid` (1 = known, 0 = unknown).

### Feature Engineering

#### Member_Age
```python
Member_Age = year(current_date()) - year(Date_of_Birth)
# NULL if DOB missing or invalid
```
**Business value:**
- Age-based segmentation (risk, pricing)  
- Compliance monitoring (minimum age)  
- Customer profiling & analytics  

#### Tenure_Days
```python
Tenure_Days = datediff(current_date(), Member_Registration_Date)
# Days as active member
```
**Business value:**
- Loyalty metrics & churn prediction  
- Retention campaign targeting  
- LTV (lifetime value) calculations  

### Write to Silver

- **Delta path:** `silverdata/members`  
- **Table:** `bupa_silver.members`  
- **Schema-aware overwrite** ensures stability  

### Metrics Logging

Tracked in `silverdata/_metrics`:
- `rowcount_members_silver`  
- `distinct_member_keys`  
- `dq_age_valid_coverage %`  
- `dq_contact_valid_coverage %`  
- `dq_address_valid_coverage %`  
- `dq_status_valid_coverage %`  
- Quarantine counts by violation type

## 5. Data Quality & Validation

### Quarantine Strategy

No bad data is silently dropped. Invalid rows are captured with:
- Violation code (PK_NULL, FK_POLICY_INVALID, AGE_INVALID, EMAIL_FORMAT, etc.)  
- Timestamp  
- Full record  

**Location:** `silverdata/_quarantine/members/<violation_type>`

### DQ Expectations

Hard rules (quarantine on failure):
- `Member_Key` not NULL  
- `Policy_ID` matches Silver Policies (FK valid)  

Soft rules (flag but include):
- `dq_age_valid`, `dq_contact_valid`, `dq_address_valid`, `dq_status_valid`  

This allows:
- Filtering in dashboards: "only DQ-clean rows"  
- Monitoring quality trends  
- Transparent governance
