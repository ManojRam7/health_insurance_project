## 1. Context & Objective

The Gold `fact_policies` layer transforms **clean Silver policies** into a **business-segmented analytics-ready fact table** that:

- 📊 Simplifies policy data for BI consumption  
- ✅ Adds business-friendly tenure, premium, discount bands  
- 💼 Creates renewal outcome labels  
- 🎯 Enables instant product & pricing analytics  
- 🔒 Maintains transparent data quality signals  

**Enterprise standards applied:**
- Fact table design (grain: one row per policy)  
- Business-friendly bucketing (Tenure, Premium, Discount)  
- Renewal outcome labeling  
- DQ flag carryover  
- Schema enforcement  
- Metrics logging  

---

## 2. Silver vs Gold fact_policies

| Area | Silver Policies | Gold fact_policies |
|------|-----------------|-------------------|
| **Grain** | One row per policy | One row per policy (enriched) |
| **Core Fields** | PK, dates, amounts | Same + business bands |
| **Tenure** | Policy_Duration_Days | Tenure_Band (0–6m, 6–12m, 1–2y, 2+y) |
| **Premium** | Annual_Premium_GBP | Premium_Band (Low/Medium/High/VH) |
| **Discount** | Discount_Offered_% | Discount_Band (buckets) + renamed field |
| **Renewal** | Flags + Conversion | Renewal_Outcome (Renewed/Not/No Offer) |
| **DQ Flags** | 3 flags (money, discount, renewal) | Same flags carried forward |
| **BI Readiness** | Intermediate | Full (bands pre-computed) |
| **Segmentation** | Limited | Rich (tenure, premium, discount, product) |

**Result:** Business-friendly fact table with pre-computed segmentations for instant analytics.

## 3. Transformations: Silver → Gold

### 3.1 Select & Rename Core Columns

**Input:** Silver Policies table

**Logic:**
```python
fact_base = silver_policies.select(
    "Policy_ID", "Customer_ID", "Product_Line", "Channel",
    "Sum_Insured_GBP", "Annual_Premium_GBP",
    "Policy_Start_Date", "Policy_End_Date", "Policy_Duration_Days",
    "Renewal_Offered_Flag", "Renewal_Accepted_Flag", "Renewal_Conversion",
    F.col("`Discount_Offered_%`").alias("Discount_Offered_Pct"),  # cleaner name
    "dq_money_valid", "dq_discount_valid", "dq_renewal_valid"
)
```

**Business impact:**
- ✅ Clean, intuitive column names  
- ✅ All core business attributes included  

### 3.2 Derive Tenure Band

**Logic:**
```python
Tenure_Band = CASE
  WHEN Policy_Duration_Days < 180 THEN "0–6 months"
  WHEN Policy_Duration_Days < 365 THEN "6–12 months"
  WHEN Policy_Duration_Days < 730 THEN "1–2 years"
  ELSE "2+ years"
END
```

**Business value:**
- 📊 Identifies new (0–6m) vs established (2+y) policies  
- 💰 Correlates with retention & profitability  
- 🎯 Supports customer lifecycle analysis  

### 3.3 Derive Premium Band

**Logic:**
```python
Premium_Band = CASE
  WHEN Annual_Premium_GBP < 250 THEN "Low (<250)"
  WHEN Annual_Premium_GBP < 750 THEN "Medium (250–750)"
  WHEN Annual_Premium_GBP < 1500 THEN "High (750–1500)"
  ELSE "Very High (1500+)"
END
```

**Business value:**
- 💰 Segments by customer value tier  
- 📈 Enables profitability analysis  
- 🎯 Drives pricing & retention strategy  

### 3.4 Derive Discount Band

**Logic:**
```python
Discount_Band = CASE
  WHEN Discount_Offered_Pct IS NULL THEN "No discount info"
  WHEN Discount_Offered_Pct = 0 THEN "No discount"
  WHEN Discount_Offered_Pct <= 10 THEN "0–10%"
  WHEN Discount_Offered_Pct <= 20 THEN "10–20%"
  ELSE "20%+"
END
```

**Business value:**
- 🏷️ Analyzes discount elasticity  
- 💼 Compares discounted vs non-discounted cohorts  
- 🎯 Informs acquisition vs profitability trade-offs  

### 3.5 Derive Renewal Outcome

**Logic:**
```python
Renewal_Outcome = CASE
  WHEN Renewal_Offered_Flag = 1 AND Renewal_Accepted_Flag = 1 THEN "Renewed"
  WHEN Renewal_Offered_Flag = 1 AND Renewal_Accepted_Flag = 0 THEN "Not renewed"
  WHEN Renewal_Offered_Flag IS NULL THEN "No renewal info"
  ELSE "No offer"
END
```

**Business value:**
- 📊 Simplifies renewal funnel analysis  
- 🎯 Identifies churn vs actively declined  
- 💼 Supports retention strategy  

### 3.6 DQ Flag Carryover

Inherit from Silver:
- `dq_money_valid` (financial sanity)  
- `dq_discount_valid` (discount bounds)  
- `dq_renewal_valid` (renewal logic)  

**Business value:**
- 🔍 Dashboards can filter to "DQ-clean" records  
- 📋 Maintains traceability  
- ✅ Transparent governance  

### 3.7 Write to Gold

- **Location:** `abfss://silverdata@clientdatastorage.dfs.core.windows.net/gold/fact_policies`  
- **Table:** `bupa_gold.fact_policies`  
- **Mode:** Overwrite with schema enforcement

## 5. Business Bands Strategy

### Why Bands Matter

Insurance analytics requires consistent segmentation across all reports. Gold bands:
- **Eliminate repeated CASE logic** in dashboards & SQL  
- **Ensure consistent KPI definitions** across teams  
- **Enable fast drill-down** analysis without pre-aggregation  

### Tenure Band Use Cases

| Band | Key Insight | Business Action |
|------|-------------|-----------------|
| **0–6 months** | New customer acquisition | Monitor onboarding, early churn |
| **6–12 months** | Early customer development | Upsell opportunities |
| **1–2 years** | Mature customer | Cross-sell, renewal focus |
| **2+ years** | Long-term customer | Loyalty programs, retention |

### Premium Band Use Cases

| Band | Key Insight | Business Action |
|------|-------------|-----------------|
| **Low (<250)** | Price-sensitive customers | Volume growth strategy |
| **Medium (250–750)** | Core customer base | Balanced retention focus |
| **High (750–1500)** | High-value customers | Dedicated support |
| **VH (1500+)** | Premium customers | VIP programs, concierge |

### Discount Band Use Cases

| Band | Key Insight | Business Action |
|------|-------------|-----------------|
| **No discount** | Full-price customers | Measure of brand strength |
| **0–10%** | Modest discounting | Standard acquisition tactic |
| **10–20%** | Aggressive discounting | Competitive market indicators |
| **20%+** | Heavy discounting | Churn mitigation or clearance |
